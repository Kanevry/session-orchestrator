## Probe 2 — reloaderoo inspect Against Our MCP Server

**Commands run (with 30s timeout each):**
```
npx reloaderoo inspect server-info -- bash scripts/mcp-server.sh
npx reloaderoo inspect list-tools  -- bash scripts/mcp-server.sh
npx reloaderoo inspect ping        -- bash scripts/mcp-server.sh
```

**Transcripts:** `03-reloaderoo-server-info.txt`, `04-reloaderoo-list-tools.txt`, `05-reloaderoo-ping.txt`

**Expected:** JSON output describing our server's protocol version, tools list, and ping response.

**Actual — server-info:**
```json
{
  "protocolVersion": "2024-11-05",
  "capabilities": { "tools": {} }
}
```
Exit 0. A benign warning ("Server does not support completions (required for completion/complete)") appears on stderr (not suppressed by `--quiet`, which appears to redirect errors to stdout instead — a minor UX quirk).

**Actual — list-tools:**
```json
{
  "tools": [
    { "name": "session_config", "description": "Reads Session Config section...", "inputSchema": {...} },
    { "name": "session_metrics", "description": "Reads the last 5 session metrics entries...", "inputSchema": {...} }
  ]
}
```
Exit 0. Two tools discovered correctly: `session_config` and `session_metrics`.

**Actual — ping:**
```json
{ "error": "MCP error -32601: Method not found: ping" }
```
Exit 0 despite error. Our MCP server (protocol version `2024-11-05`) does not implement the `ping` method. This is a server-capability gap, not a reloaderoo bug.

**Key observations:**
- `--quiet` flag behavior is unexpected: it does NOT suppress the completions-support warning; instead the error appears on stdout in the same output stream. The verbose (non-quiet) run shows JSON on stdout + warning on stderr separately. The `--quiet` flag appears to collapse all output to stdout.
- `call-tool` was not tested (requires tool parameters) but the schema for both tools shows no required inputs — `call-tool session_config` would be valid.
- Server correctly identified 2 tools with full `inputSchema` shapes.

**Status: PASS (with one caveat).** Core inspect functionality works against our MCP server. `ping` fails with -32601 because `mcp-server.sh` uses protocol `2024-11-05` which predates the ping spec — acceptable, not a blocker. The `--quiet` flag behavior diverges from documented "raw JSON only" — worth noting in the ADR.

## Probe 3 — Stop-Hook Verification Stub End-to-End

**Approach:** Wrote a minimal `stop-verify-stub.mjs` to `/tmp/` (via Write tool — not production code) implementing the full A5 contract. Ran 4 deterministic sub-cases.

**Transcript:** `06-stop-verify-stub.txt`

**Sub-cases and results:**

| Case | Payload | Expected exit | Actual exit | Pass? |
|------|---------|---------------|-------------|-------|
| 3a | `stop_hook_active=true`, cmd=`false` | 0 (safety gate) | **0** | PASS |
| 3b | `stop_hook_active=false`, cmd=`true` | 0 (verify pass) | **0** | PASS |
| 3c | `stop_hook_active=false`, cmd=`false` | 2 (block stop) | **2** | PASS |
| 3d | `stop_hook_active=true`, cmd=`false` | 0 (safety gate overrides) | **0** | PASS |

**All 4 cases passed.** The pattern is fully implementable in Node.js ESM using only `node:fs` (`readSync`) and `node:child_process` (`spawnSync`) — zero external dependencies.

**Key observations:**
- `readSync(0, ...)` in a tight loop correctly drains stdin before JSON parse — handles both small and large payloads.
- `spawnSync` with `timeout: 60000` provides wall-time cap for the verification command.
- stderr output from the stub (`[stop-verify-stub] ...` lines) would flow into Claude's transcript as required by the A5 spec — this is the "failure evidence" mechanism.
- The `stop_hook_active` safety gate fires at line 1 of logic, before any command is dispatched — correct per Anthropic contract.
- One issue: stderr output from `spawnSync` is empty in case 3c because `false` has no stderr. A real verification command (e.g., `npm test`) would populate this field with actionable failure output.

**Status: PASS.** The design is proven end-to-end. Ready for `hooks/on-stop.mjs` integration.

## Probe 4 — events.jsonl Shape + Size

**Command run:**
```
tail -10 .orchestrator/metrics/events.jsonl
```

**Transcript:** `07-events-jsonl-tail.txt`

**Actual output (last 10 entries):**
```jsonl
{"event":"stop","timestamp":"2026-05-10T14:18:59.220Z","session_id":"bed53b02...","wave":0,"branch":"main","commit":"0c74f1fb...","duration_ms":0}
{"event":"subagent_stop","timestamp":"2026-05-10T14:22:36.116Z","agent":"Explore"}
... (8 more subagent_stop entries)
```

**File stats:**
- **Lines: 1,425** (not 86K as estimated in A6 — that was a cross-instance artifact)
- **Size: 184 KB**
- **Event type distribution:** `subagent_stop`=911, `stop`=362, `orchestrator.session.started`=135, test/noise entries=16

**Schema fields observed:**
- `stop` events: `session_id`, `wave`, `branch`, `commit`, `duration_ms`, `timestamp`
- `subagent_stop` events: `agent`, `timestamp` (minimal — no `wave` or `session_id` linkage)

**Key observations:**
- The `subagent_stop` entries lack `session_id` and `wave` fields — this is a schema gap that would complicate failure tracing if `failures.jsonl` (proposed in PRD #366) needs to correlate failures back to a specific wave/session.
- `stop` events have `duration_ms: 0` — the duration field is not being populated. The hook fires immediately without computing elapsed time.
- File is 184 KB at 1,425 lines. At current growth rate (~135 sessions, ~10 events/session average), the retention concern from A6 is valid but not yet critical.

**Status: PASS.** File is readable, shape is confirmed, gaps documented.

## Findings & Feedback for ADRs / PRD

### What WORKED — gives ADR/PRD authors confidence

1. **reloaderoo 1.1.5 installs clean via npx, zero config required.** Adopt-directly decision is validated.
2. **`inspect list-tools` returns correct, parseable JSON** against our `mcp-server.sh` in under 5 seconds. MCP introspection tooling is viable for `mcp-builder` skill docs and CI smoke tests.
3. **`--quiet` + `inspect` is a usable combination** for scripted JSON extraction (with the caveat that stderr warnings are mixed into stdout — use `2>/dev/null` in automation).
4. **Stop-hook safety gate (`stop_hook_active` check) works correctly** in all 4 edge cases including the double-block scenario (3d). The design from A5 is ready to implement.
5. **Stop-hook exit code 2 correctly blocks** — this is the mechanism that forces continuation; it is confirmed to work in Node.js ESM with `process.exit(2)`.
6. **events.jsonl schema is stable and readable** — the `stop` event has sufficient fields (`session_id`, `wave`, `commit`) to anchor a `failures.jsonl` cross-reference.

### What FAILED or DID NOT WORK AS EXPECTED — must be captured in ADR risks

1. **`reloaderoo inspect ping` returns `MCP error -32601: Method not found`** against our server. Our `mcp-server.sh` uses protocol `2024-11-05` which predates the ping spec. Any ADR recommending ping-based health checks must note this protocol version dependency.
2. **`--quiet` flag does not suppress the completions-support warning** — it appears on stdout, not stderr. Automation scripts using `reloaderoo --quiet inspect` must pipe through `grep -v "Error: Server does not support"` or use `2>/dev/null`. Document in `skills/mcp-builder/SKILL.md`.
3. **`subagent_stop` events lack `session_id` and `wave`** — the minimal schema prevents correlating subagent failures with their parent wave. PRD #366's `failures.jsonl` design must either enrich `subagent_stop` events or rely on timestamp proximity for correlation.
4. **`duration_ms: 0` in stop events** — the on-stop hook is not computing elapsed time. The proposed failures.jsonl needs its own timing logic; it cannot reuse events.jsonl duration.
5. **Stop-hook stderr evidence is empty for shell builtins** — `false` has no stderr output. ADR must specify that verification commands should produce actionable stderr (e.g., `npm test -- --reporter verbose`) to make the Claude-context feedback loop useful.
